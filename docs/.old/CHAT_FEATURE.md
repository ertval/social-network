## Overview

The chat feature provides **real-time one-to-one direct messaging** between users. It is built on:

- A **WebSocket** connection for real-time message exchange (send, receive, history, mark-read).
- Two **REST endpoints**: `POST /api/v1/chat/init` to open or retrieve a conversation by user ID, and `GET /api/v1/chat/users` to populate the chat user list with online presence and last-message ordering.
- A **SQLite** storage layer with three tables: `direct_chats`, `chat_messages`, and `chat_reads`.

There is no concept of group chats — every chat is strictly between **two** users.

## Database Schema

### `direct_chats`

Represents a unique conversation channel between two users. The pair is stored in a normalized form: the user with the lexicographically smaller ID is always `user_low_id`.

```sql
CREATE TABLE direct_chats (
    id              TEXT PRIMARY KEY,           -- UUID generated by server
    user_low_id     TEXT NOT NULL,              -- smaller UUID alphabetically
    user_high_id    TEXT NOT NULL,              -- larger UUID alphabetically
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_message_id INTEGER,                    -- FK → chat_messages.id (nullable)
    last_message_at DATETIME,                   -- nullable, for sorting
    UNIQUE(user_low_id, user_high_id)           -- one chat per pair, enforced by DB
);
```

**Why low/high normalization?**
Storing the user pair as `(low, high)` instead of `(a, b)` or `(b, a)` means there is only one possible row per pair. `GetOrCreateChat` uses `INSERT OR IGNORE`, so a second attempt to create the same chat silently returns the existing one.

---

### `chat_messages`

```sql
CREATE TABLE chat_messages (
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    chat_id           TEXT NOT NULL REFERENCES direct_chats(id) ON DELETE CASCADE,
    sender_id         TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    content           TEXT NOT NULL,
    client_message_id TEXT,                     -- optional dedup key from client
    created_at        DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(sender_id, client_message_id)        -- idempotency guard
);
```

**`client_message_id`** is an optional, client-supplied string. If the client sends the same message twice with the same `client_message_id`, the second `INSERT` is rejected by the unique constraint, preventing duplicate messages on reconnect or retry. If not needed, simply omit it from the payload.

---

### `chat_reads`

Tracks per-user read state within each chat. One row per `(chat_id, user_id)`.

```sql
CREATE TABLE chat_reads (
    chat_id              TEXT NOT NULL REFERENCES direct_chats(id) ON DELETE CASCADE,
    user_id              TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    last_read_message_id INTEGER REFERENCES chat_messages(id) ON DELETE SET NULL,
    last_read_at         DATETIME,
    unread_count         INTEGER DEFAULT 0,
    updated_at           DATETIME DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (chat_id, user_id)
);
```

- **`unread_count`** is incremented atomically on the **recipient's** row whenever a new message is sent (done in the same transaction as the message insert in `SendMessage`).
- **`MarkAsRead`** resets `unread_count` to `0` and records `last_read_message_id`.

### `chat.Repository` interface

All database operations go through this interface:

| Method                                                          | Description                                                                                    |
| --------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `GetOrCreateChat(ctx, userID1, userID2)`                        | Idempotent — creates or fetches the chat between two users                                     |
| `GetChat(ctx, chatID)`                                          | Fetch a single chat by its ID                                                                  |
| `GetChatsForUser(ctx, userID)`                                  | All chats for a user, sorted by `last_message_at DESC`                                         |
| `SendMessage(ctx, chatID, senderID, content, clientMessageID)`  | Insert message + update `direct_chats` + increment recipient unread count (single transaction) |
| `GetMessagesForChat(ctx, chatID, limit)`                        | Latest N messages (newest first)                                                               |
| `GetMessagesForChatBefore(ctx, chatID, beforeMessageID, limit)` | Paginate older messages before a cursor                                                        |
| `MarkAsRead(ctx, chatID, userID, upToMessageID)`                | Reset unread count to 0, record last read message                                              |
| `GetUnreadCount(ctx, chatID, userID)`                           | Single chat unread count for a user                                                            |
| `GetAllUnreadCounts(ctx, userID)`                               | Map of `chatID → unreadCount` for all chats with unread messages                               |

## WebSocket Protocol

### Connection & Authentication

**Endpoint:** `GET /api/v1/ws`
**Protocol upgrade:** HTTP → WebSocket (via `gorilla/websocket`)
**Auth:** Required — uses the same `access_token` cookie as all other protected routes.

The `Authorization.Required` middleware runs first and validates the session. On success it stores the authenticated `*user.User` in the request context. The WebSocket upgrade handler then reads the user from that context — no separate cookie parsing happens inside the WebSocket layer.

**The `access_token` cookie must be present when establishing the WebSocket connection.** If the browser has a valid session, the cookie is sent automatically. No extra auth steps are needed after the connection is open.

```
GET ws://host/api/v1/ws
Headers:
  Cookie: access_token=<token>
  Upgrade: websocket
  Connection: Upgrade
```

Once connected, the client is registered in the **Hub** under their `userID`. A user can have multiple simultaneous connections (e.g., multiple browser tabs) — all will receive messages sent to that user.

---

### Keep-Alive (Ping/Pong)

The server sends a WebSocket **ping frame** every **54 seconds** (`pingPeriod = (60s * 9) / 10`). The client must respond with a pong (browsers handle this automatically for native WebSocket). If no pong is received within **60 seconds**, the server closes the connection.

Additionally, the application layer supports explicit JSON pings:

**Client → Server:**

```json
{ "type": "ping", "request_id": "optional-id" }
```

**Server → Client:**

```json
{ "type": "pong", "request_id": "optional-id" }
```

This is separate from the WebSocket protocol-level ping and can be used to test round-trip latency from JavaScript.

---

### Message Envelope Format

Every message in both directions is a JSON object with this envelope:

```json
{
  "type": "string",
  "request_id": "optional-string",
  "payload": {}
}
```

| Field        | Description                                                                                                     |
| ------------ | --------------------------------------------------------------------------------------------------------------- |
| `type`       | Identifies the message type (see tables below)                                                                  |
| `request_id` | Optional. Client-supplied identifier echoed back in the response. Use this to correlate request/response pairs. |
| `payload`    | A JSON object specific to the message type                                                                      |

The maximum message size the server accepts is **4096 bytes** per frame.

---

### Inbound Message Types (Client → Server)

#### `ping`

```json
{
  "type": "ping",
  "request_id": "r1"
}
```

No payload required. Responds with `pong`.

---

#### `chat.send` — Send a message

```json
{
  "type": "chat.send",
  "request_id": "r2",
  "payload": {
    "chat_id": "uuid-of-chat",
    "content": "Hello!",
    "client_message_id": "unique-client-id"
  }
}
```

| Field               | Required | Description                                                                      |
| ------------------- | -------- | -------------------------------------------------------------------------------- |
| `chat_id`           | **Yes**  | The UUID of the `direct_chats` row                                               |
| `content`           | **Yes**  | Message text (must not be empty)                                                 |
| `client_message_id` | No       | Dedup key. If you retry on reconnect, reuse the same value to prevent duplicates |

**Server response** (type `chat.message`):

- Sent to the **sender** as confirmation.
- Also sent to the **recipient** if they have an active WebSocket connection.

---

#### `chat.history` — Load message history

```json
{
  "type": "chat.history",
  "request_id": "r3",
  "payload": {
    "chat_id": "uuid-of-chat",
    "before_message_id": 0,
    "limit": 10
  }
}
```

| Field               | Required | Description                                                                                                           |
| ------------------- | -------- | --------------------------------------------------------------------------------------------------------------------- |
| `chat_id`           | **Yes**  | The UUID of the chat                                                                                                  |
| `before_message_id` | No       | Pagination cursor. Omit (or set `0`) for the latest messages. Set to the oldest `id` you have to load older messages. |
| `limit`             | No       | Number of messages to return. Default `10`, max `20`.                                                                 |

**Messages are returned in reverse-chronological order (newest first).** The frontend should reverse them for display.

---

#### `chat.mark_read` — Mark messages as read

```json
{
  "type": "chat.mark_read",
  "request_id": "r4",
  "payload": {
    "chat_id": "uuid-of-chat",
    "up_to_message_id": 42
  }
}
```

| Field              | Required | Description                                                         |
| ------------------ | -------- | ------------------------------------------------------------------- |
| `chat_id`          | **Yes**  | The UUID of the chat                                                |
| `up_to_message_id` | **Yes**  | The highest message ID the user has seen. Resets unread count to 0. |

No response payload is sent on success (the connection stays open). On error, an `error` message is sent.

---

### Outbound Message Types (Server → Client)

#### `chat.message` — New message delivered

Sent in two scenarios:

1. Back to the **sender** as confirmation after `chat.send`.
2. To the **recipient** in real time if they are online.

```json
{
  "type": "chat.message",
  "request_id": "r2",
  "payload": {
    "id": 42,
    "chat_id": "uuid-of-chat",
    "sender_id": "uuid-of-sender",
    "content": "Hello!",
    "created_at": "2026-04-29T10:30:00Z"
  }
}
```

Note: `request_id` is only present on the sender's confirmation. The push to the recipient has the same `request_id` echoed from the sender's original request.

---

#### `chat.history_result` — History response

```json
{
  "type": "chat.history_result",
  "request_id": "r3",
  "payload": [
    {
      "id": 42,
      "chat_id": "uuid-of-chat",
      "sender_id": "uuid-of-sender",
      "content": "Hello!",
      "created_at": "2026-04-29T10:30:00Z",
      "client_message_id": null
    },
    { ... }
  ]
}
```

`payload` is a JSON array of message objects (newest first). `client_message_id` is `null` when not set.

---

#### `error` — Error response

```json
{
  "type": "error",
  "request_id": "r2",
  "payload": {
    "message": "chat_id and content are required"
  }
}
```

Common error messages:

| Message                              | Cause                                      |
| ------------------------------------ | ------------------------------------------ |
| `"invalid message format"`           | Malformed JSON envelope                    |
| `"invalid payload"`                  | Payload JSON does not match expected shape |
| `"chat_id and content are required"` | Missing required fields in `chat.send`     |
| `"failed to send message"`           | Database error during insert               |
| `"failed to load messages"`          | Database error during select               |
| `"failed to mark as read"`           | Database error during upsert               |
| `"unknown message type"`             | `type` field not recognized                |

---

#### `pong`

```json
{
  "type": "pong",
  "request_id": "r1"
}
```

## REST Endpoints

### `POST /api/v1/chat/init`

**Auth:** Required

Opens or retrieves the direct chat between the authenticated user and another user. This is the **first call to make** before any WebSocket messaging — it returns the `chat_id` required by all WebSocket operations. The operation is idempotent: calling it multiple times for the same pair always returns the same chat.

**Request body:**

```json
{ "user_id": "uuid-of-the-other-user" }
```

| Field     | Required | Description                                                        |
| --------- | -------- | ------------------------------------------------------------------ |
| `user_id` | **Yes**  | The ID of the user you want to chat with. Must not be your own ID. |

**Response `200 OK`:**

```json
{
  "info": null,
  "data": {
    "ID": "uuid-of-chat",
    "UserLowID": "uuid",
    "UserHighID": "uuid",
    "CreatedAt": "2026-04-29T10:00:00Z",
    "UpdatedAt": "2026-04-29T10:00:00Z",
    "LastMessageID": null,
    "LastMessageAt": null
  }
}
```

The `data.ID` field is the `chat_id` to use for all subsequent WebSocket messages.

**Error responses:**

| Status | Reason                                         |
| ------ | ---------------------------------------------- |
| `400`  | Missing `user_id`, or `user_id` is your own ID |
| `401`  | Not authenticated                              |
| `500`  | Database error                                 |

---

### `GET /api/v1/chat/users`

**Auth:** Required

Returns all registered users (except yourself), enriched with online presence and chat ordering data.

**Response `200 OK`:**

```json
{
  "info": null,
  "data": [
    {
      "user_id": "uuid",
      "nickname": "alice",
      "is_online": true,
      "last_message_at": "2026-04-29T10:30:00Z"
    },
    {
      "user_id": "uuid2",
      "nickname": "bob",
      "is_online": false,
      "last_message_at": null
    }
  ]
}
```

**Sorting rules:**

1. Users you have **previously chatted with** (`last_message_at != null`) come first, sorted by most recent message (newest first — mirrors Discord).
2. Users you have **never chatted with** (`last_message_at == null`) come after, sorted alphabetically by `nickname`.

**`is_online`** is determined by whether the user currently has an active WebSocket connection to the server. This is a real-time value at the moment of the HTTP request.

> **Note:** `is_online` via this HTTP endpoint is only accurate at request time. For live presence updates you would need a WebSocket event (not yet implemented).

## Full Workflow — Step by Step

### Step 1: Authenticate

Log in via `POST /api/v1/login/email` or `/login/username`. A `session_token` cookie is set automatically. All subsequent requests (HTTP and WebSocket) use this cookie.

### Step 2: Open the chat panel / load user list

```
GET /api/v1/chat/users
```

Render the returned user list sorted as described above. Each item gives you `user_id`, `nickname`, `is_online`, and `last_message_at`.

### Step 3: Connect WebSocket

```javascript
const ws = new WebSocket('ws://localhost:8080/api/v1/ws');
// No extra headers needed — the session cookie is sent automatically
```

Once connected, you can start sending and receiving messages.

### Step 4: Open a conversation with a user

Call `POST /api/v1/chat/init` to get the `chat_id` for the conversation. This is idempotent — call it every time the user clicks on someone in the chat list.

```javascript
const res = await fetch('/api/v1/chat/init', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ user_id: targetUserID }),
});
const { data } = await res.json();
const chatID = data.ID; // use this for all WebSocket messages
```

### Step 5: Load recent messages

```javascript
ws.send(
  JSON.stringify({
    type: 'chat.history',
    request_id: 'hist-1',
    payload: {
      chat_id: 'uuid-of-chat',
      limit: 10,
    },
  })
);
```

Handle the `chat.history_result` response and reverse the array before rendering (it arrives newest-first).

### Step 6: Send a message

```javascript
ws.send(
  JSON.stringify({
    type: 'chat.send',
    request_id: 'send-1',
    payload: {
      chat_id: 'uuid-of-chat',
      content: 'Hey, how are you?',
      client_message_id: crypto.randomUUID(), // recommended for dedup
    },
  })
);
```

Listen for `chat.message` responses. The same `chat.message` payload is pushed to the recipient if they are online.

### Step 7: Receive messages from others

Handle incoming messages while the WebSocket is open:

```javascript
ws.onmessage = (event) => {
  const envelope = JSON.parse(event.data);
  if (envelope.type === 'chat.message') {
    const msg = envelope.payload;
    // msg.chat_id, msg.sender_id, msg.content, msg.created_at, msg.id
    appendMessageToUI(msg);
  }
};
```

### Step 8: Mark messages as read

Call this when the user opens a chat, scrolls to the bottom, or the window gains focus:

```javascript
ws.send(
  JSON.stringify({
    type: 'chat.mark_read',
    request_id: 'read-1',
    payload: {
      chat_id: 'uuid-of-chat',
      up_to_message_id: 42, // highest message.id the user has seen
    },
  })
);
```

### Step 9: Load older messages (pagination)

When the user scrolls to the top, request earlier messages:

```javascript
ws.send(
  JSON.stringify({
    type: 'chat.history',
    request_id: 'hist-2',
    payload: {
      chat_id: 'uuid-of-chat',
      before_message_id: 10, // oldest message.id currently shown
      limit: 10,
    },
  })
);
```

Prepend the returned messages to the top of the chat view.

## Concurrency & Online Presence

### Hub

The `Hub` is a single, server-wide singleton started when the HTTP server starts. It holds a map of `userID → set of *Client` connections protected by a `sync.RWMutex`.

```
Hub.clients = map[string]map[*Client]bool
                     │
                  userID  →  { conn1: true, conn2: true, ... }
```

- **Register:** Called when a new WebSocket connection is upgraded. Multiple connections per user are supported (e.g., two browser tabs).
- **Unregister:** Called automatically when a connection closes (via `ReadPump` defer).
- **Send:** Delivers a `[]byte` message to **all** active connections of a given `userID`.
- **IsOnline:** Returns `true` if the user has at least one active connection.

### Client goroutines

Each `*Client` runs two goroutines:

| Goroutine   | Role                                                                            |
| ----------- | ------------------------------------------------------------------------------- |
| `ReadPump`  | Reads frames from the WebSocket, calls `onMessage`, handles close/error         |
| `WritePump` | Drains the `send` channel (buffered, size 256), sends frames, sends ping frames |

The `send` channel being buffered means slow clients do not block message delivery to other users.

## Unread Counts

Unread counts are managed automatically by the database layer:

1. **On message send** (`SendMessage`): The `chat_reads` row for the **recipient** has `unread_count` incremented by 1 atomically (upsert in same transaction as the message insert). The sender's unread count is unaffected.
2. **On mark-read** (`MarkAsRead`): The `chat_reads` row for the **requesting user** has `unread_count` reset to `0` and `last_read_message_id` updated.
3. **Reading counts** (`GetUnreadCount`, `GetAllUnreadCounts`): These return counts from `chat_reads`. If no row exists for a `(chat_id, user_id)` pair, the count is `0`.

`GetAllUnreadCounts` returns a `map[chatID]unreadCount` for all chats where `unread_count > 0`. This can be used to render badge counts on the chat user list or a navbar indicator.

> **Note:** `GetAllUnreadCounts` is available in the repository but is **not yet exposed via any endpoint**. A frontend developer wanting to show aggregate unread badges will need a new HTTP endpoint wired to this method.

---

## Pagination

Messages use **cursor-based pagination** by message ID:

- **Latest page:** Send `chat.history` with no `before_message_id` (or `0`). Returns the latest `limit` messages.
- **Next (older) page:** Send `chat.history` with `before_message_id` set to the smallest `id` from the previous batch.
- **Direction:** Results are always in **descending order** by `created_at` (newest first). The frontend must reverse the array for chronological display.
- **Limit:** Default `10`, max `20`. Values outside this range are clamped to `10`.

```
Page 1:  history(chat_id, limit=10)       → msgs [50..41]
Page 2:  history(chat_id, before=41, 10)  → msgs [40..31]
Page 3:  history(chat_id, before=31, 10)  → msgs [30..21]
...
```

## Key Design Decisions

| Decision                                               | Rationale                                                                                                                                                                          |
| ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Normalized user pair (low/high IDs)**                | Ensures one unique row per pair in `direct_chats` without needing to query both orderings.                                                                                         |
| **`INSERT OR IGNORE` for chat creation**               | Makes `GetOrCreateChat` idempotent at the database level — safe to call concurrently.                                                                                              |
| **`UNIQUE(sender_id, client_message_id)` on messages** | Client-side dedup: retrying a send on reconnect with the same `client_message_id` will not create a duplicate message.                                                             |
| **Single transaction for send + unread increment**     | The message insert and the recipient's unread count update are atomic — no partial state if either fails.                                                                          |
| **Hub with `sync.RWMutex`**                            | Allows many concurrent readers (message delivery to online users) with exclusive write locks only for register/unregister events.                                                  |
| **`send` channel buffered to 256**                     | WritePump is non-blocking from the Hub's perspective; slow network clients do not stall delivery to others.                                                                        |
| **Session cookie for WS auth**                         | Reuses the same auth infrastructure as all HTTP routes. The cookie is validated once during the HTTP upgrade handshake; no token exchange inside the WebSocket protocol is needed. |
| **Messages returned newest-first**                     | Simplifies pagination (cursor = smallest ID seen). The frontend reverses for display.                                                                                              |
