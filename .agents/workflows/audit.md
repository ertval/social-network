Perform a structured, multi-agent code review of the changes introduced in this pull request.

Since this repository enforces strict boundaries, architectural paradigms, and performance standards, please proceed through the following execution phases:

---

## 🛠️ Execution Phases

### Phase 1: Deterministic Grounding & Tool Gates
Before spawning any cognitive subagents, verify code health using deterministic tooling.
 * Run full CI pipeline (BE + FE): `rtk make ci`
 * **Early Exit**: If any step fails, compile the errors and output a failing report immediately. Do not trigger Phase 2.

### Phase 2: Cognitive Subagent Review
If deterministic tools pass, launch specialized subagents using the guidelines in `.agents/prompts/audit.md`:
1. **Ticket & Scope Drift Agent**: Scans for out-of-scope code, unused functions/variables (making sure your changes didn't create dead code), and requirements alignment. PR size. **Verifies that `docs/sprints/ticket-tracker.md` was updated to mark the target ticket as completed.**
2. **Logic & Correctness Agent**: Validates business logic, checks for database connection pooling (1-10 connections max), WAL mode configurations, and busy timeouts.
3. **Architecture & Boundary Rules Agent**: Verifies vertical slice rules (`internal/<slice>`), D5 boundary rules (no cross-slice transport/store imports), and D3 ID-only communications/Event Bus mutations.
4. **Code Quality, Security & Framework Best Practices Agent**: Checks for SQL injections, WebSocket timeouts/limits, unauthorized dependency updates, and Next.js / Go structured logging conventions.
5. **Testing & Migration Integrity Agent**: Checks Go table-driven tests, store tests isolation using independent SQLite databases, and DB migration sequencing.
6. **Frontend & UI Compliance Agent**: Checks React Server Component (RSC) vs Client Component boundaries, F1 audit checklist feature mapping, client-side file/image uploads & size limit validations, real-time WebSocket heartbeats, follow-based chat authorization gates, and cookie-based session persistence.

### Phase 3: Adversarial Validation (Judge Pass)
Review findings generated by the subagents:
1. Act as a Judge to verify findings against the actual file contents at specified lines.
2. Remove any false positives or duplicates.
3. Verify that the recommendations are optimal, noting style consistency.

### Phase 4: Synthesis & Output Report
Generate the final markdown report and save it to `docs/reviews/PR_REVIEW_REPORT.md` following the schema outlined in `.agents/prompts/audit.md`.

Provide a summary of the report to the developer with actionable next steps to pass the review.
